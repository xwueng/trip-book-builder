TRIP BOOK RULES
(this file: trip_book_spec.md — attached to each student's Claude Project; tells
Claude how to build the student's own flip-through trip book. Separate from
instructions_spec.md, which covers the Instructor_Guide.html / Student_Handbook.html
class-facilitation guides.)

Build a flip-through trip book from uploaded photos: one self-contained HTML file, phone-friendly, real page-turning flip. No music unless asked.

Design: cream paper w/ soft vignette, cream-white cards, teal headings, amber accent, warm dark ink. Serif display font (Palatino/Book Antiqua/Georgia) for title + place names, sans-serif for body text. Cover/closing: thin bordered frame. Cover = tracked-uppercase kicker ("A Trip Book"), large serif title, wavy divider with a dot at each end, dates (or placeholder), "Tap Next to begin" hint, illustrated postmark stamp top corner (dashed border, inner frame, circle motif, landscape silhouette, short label) — not plain text. Closing = same frame, serif "The End", route line of places joined by "·", short closing line. When the closing page includes a map (per the Map rule below), give the map the space: shrink "The End" and tighten the spacing above it (top-aligned rather than vertically centered, small title, minimal divider margin) rather than centering a large title over a small map — the map is the more useful content and should be the largest thing on the page, not squeezed into leftover space beneath a big heading. Fact callout: tinted box, left amber border, sentence only — no "Fact"/"Fun Fact" label. Nav bar: "Back"/"Next" fixed to bottom, gradient fade above, safe-area padding; fixed-size (not stretched) chunky/tactile buttons with 3D-press feel; counter centered between them.

Structure: cover → one page per photo (send order) → closing page (+ map on closing, whenever the Map rule below produces at least one plotted stop). Each photo page, in this exact order top to bottom: photo, place, date, note, then the fact callout. The fact box must come after the note in the actual markup, not just visually — a fact box placed earlier in the DOM (e.g. right after the photo) with `margin-top:auto` to visually push it down is fragile: it only looks right by accident, and any variation in note length either overlaps content or leaves large dead space. Give the fact box a small fixed top margin instead, so it sits immediately after the note regardless of note length. No "Stop 1/2/3" labels.

Page turn: hinge-and-curl from the edge, like a real page turning. Never a flat rotate/spin. Hinge stays on the same spine edge (left) for every page, cover to closing — set transform-origin to the left edge, not the right; a right-anchored hinge can look plausible flipping forward (the page vanishes past 90° once backface-visibility hides it) but is immediately visible as wrong when flipping backward, so test the backward direction specifically, not just forward. Stacking order: turned pages and not-yet-turned pages must be tracked in two non-overlapping z-index ranges (e.g. turned = page index, unturned = a large constant minus page index) — never a single formula like `total - index` for the unturned side, since that range collides with the turned range partway through any book longer than a few pages and makes later pages flip from the wrong side. During the flip itself, the actively-turning page must stay above the entire stack for the full duration of its transition (so it visibly arcs over the book), and only drop to its settled low z-index after the transition finishes (e.g. via a timeout matching the CSS transition duration) — assigning its final low z-index at the same instant the flip starts makes it sink behind the stack mid-animation instead of passing over it. Flip-count off-by-one: the number of possible flips is (leaf count − 1), not the leaf count — the last leaf never needs to flip forward. Guard `next()` and disable the Next button against that `maxTurn` value, not against total leaf count; using leaf count for both directly causes the Next button to never disable on the closing page, and a further click will try to flip the closing leaf itself. Check the flip at the book's midpoint and near the end, not just the first couple of pages, before delivering — and specifically click Next once more after reaching the closing page to confirm it's actually disabled.

Memory footprint for the flip mechanic: every leaf lives permanently in the DOM for the whole session (that's what makes the 3D flip work — a leaf has to be there to flip back to), so anything that keeps all of them fully resident in memory at once scales badly as photo count grows. Two specific traps to avoid: (1) Never set `will-change: transform` permanently on the `.leaf` class in CSS — combined with `preserve-3d`, that forces every leaf into its own separate GPU-composited texture for the entire session, not just the one being animated. Apply `will-change` dynamically via JS only to the leaf currently mid-flip, and clear it back to `auto` once that flip settles (in the same `settle()` step that resets z-index). (2) Don't embed every photo's data URI directly in `src` — put it in `data-src` instead, and have JS populate the real `src` only for the current leaf and a small window of neighbors (e.g. current ± 2), clearing `src` on leaves outside that window as the person navigates. Without this, a 20-photo book can hold ~100MB of decoded image data resident simultaneously regardless of which page is visible, which is a plausible cause of a memory-pressure crash on a constrained mobile WebView (symptoms: the book goes blank/dark and resets to the cover, or flashes repeatedly) — and pinch-zooming, which forces a full re-rasterization of everything at once, is a very plausible trigger for tipping it over. Keep the viewport meta tag simple (`width=device-width, initial-scale=1.0`) and do not disable pinch-zoom to work around this — people reasonably expect to be able to zoom into a photo, blocking the gesture doesn't address the underlying memory cost, and doing so was tried and confirmed not to fix the crash while removing a feature people wanted. Note: even with both fixes applied, a zoom-triggered flash/reset has been reported as still occurring — treat this as a known, only partially-understood risk on memory-constrained WebViews rather than a solved problem, keep total embedded image payload as small as reasonable (resize/compress before embedding), and don't claim this is fixed without the person confirming on their actual device.

3D rendering — cross-browser: every property driving the flip (`perspective`, `transform-style`, `backface-visibility`, and the `transform` that rotates each leaf, both in CSS and wherever JS sets `.style.transform`) needs its `-webkit-` prefixed form set alongside the standard one. Chromium accepts the unprefixed versions and will look correct in an in-house check, but Safari/iOS WebKit (the realistic delivery target for a phone-book) often still requires the prefixed form for `backface-visibility` and `perspective` specifically — without it, a leaf's back face can bleed through as a near-blank ghost card behind the active page. Don't treat a clean Chromium screenshot as sufficient proof this works; the failure mode is invisible in Chromium and only shows up on WebKit.

Sizing: the book must scale to fill whichever dimension — viewport width or height — is actually the tighter constraint (e.g. `max-width: min(94vw, 92vh * <aspect-based factor>)`), not stop at a fixed pixel cap. A fixed max-width that's comfortable on an actual phone screen still leaves a large expanse of the page background exposed on any wider viewport (a laptop browser, a wide preview pane), which reads as an unwanted frame or border around the book. Verify this by rendering at more than one viewport size before delivering, not just the narrow phone width — the gap is invisible at phone width and only shows up wider.

Uploads: build immediately when photos are sent, even with no caption or text. Photos-only send is enough to trigger the build — don't wait for a description.

Dates: show only if in metadata or given; otherwise omit, no placeholder. Missing metadata is normal (uploads often strip GPS/date on export) — not an error.

Places/facts: identify each photo's place. Confident ID → state plainly. Guess → still write a fact, and flag with a "confirm venue" badge on the photo's corner. This guessed/confirmed flag is a single fact about the place and must drive both the photo page's badge and the closing map's marker styling from the same source — don't set one without the other. When editing either the photo pages or the map separately (e.g. a later map-only revision), re-check that badge count and guessed-marker count still match; they drifted apart once already in this project (a map showing 4 guessed stops while zero photo pages actually carried the badge) because the map was rebuilt without touching the photo pages it was describing.

Map: include a static map on the closing page whenever at least one photo has an identified place — confirmed or guessed. Do not require every location to be confirmed before showing a map; plot every identified stop, confirmed and guessed together. Multiple photos at the same place/city collapse to one marker unless the map is zoomed in tight enough to separate them. Only omit the map entirely if no photo in the book has any identified place at all (nothing to plot). Re-evaluate on every rebuild — a later caption, confirmation, or new upload can add stops or convert a guessed marker to confirmed. If the book's photos span multiple distant regions or countries, offer a choice: one country/region-scale overview (stops may compress or cluster at that zoom) vs. closer-in maps per region — don't silently pick one.

Map style — sourcing: never hand-draw land, coastline, or lake shapes from memory or by eyeballing curves; freehand bezier paths default to smooth round "blob" shapes that don't read as real geography. Instead, source real shape data — search for and fetch actual coastline/lake/boundary geometry (e.g. Natural Earth or similar datasets hosted on GitHub, fetched via raw.githubusercontent.com) and real latitude/longitude for each landmark — then project it onto the canvas with a latitude-corrected equirectangular projection (scale longitude by cos(mean latitude) so shapes aren't stretched). Every marker's position comes from this same projection, not eyeballed placement. Never call a live map-tile or geocoding API at runtime (no external map service is contacted while the book is open; this keeps the trip book a single self-contained file with no external dependencies, matching the rest of the spec) — the real data is fetched once while building the map, then baked into static SVG.

Map style — drawing: cream/white land, blue water, green park patches, gray roads with white centerlines, illustrative touches where they fit (a small bridge/peak/building sketch, a ferry line, etc.). Markers are plain colored numbered circles — not teardrop pins — each a distinct color (cycle through a fixed palette if there are more stops than colors), with the number in white. No legend box; instead, a text label in the marker's own color sits directly beside it, so the label itself is the legend. Guessed stops (the same ones flagged "confirm venue" on their photo page) get a dashed ring around their marker in addition to their normal color, so a guessed stop never carries the same visual confidence as a confirmed one. Compass arrow always included; a scale bar is included at regional/city scale and may be omitted at country/multi-region overview scale, where a single linear scale reads as less meaningful. Include a route line in send order only when it adds clarity (e.g. a walking/driving route) — for a country-scale overview of unrelated stops, markers and labels alone are usually enough.

Map style — layout and verification: when a marker's true coordinate would collide with another marker, a decorative icon, or text, don't place the numbered circle there — offset it to a nearby clear spot and connect it back to a small dot at the true coordinate with a thin leader line in the marker's color. Search for label positions the same way: don't guess a spot and assume it's clear. Before presenting any map, run an actual geometric overlap check (e.g. via shapely) across every element that was placed — land/water shapes, every marker, every leader line, every label, compass, scale bar — confirming zero unexpected overlaps and everything within canvas bounds. Fix and re-run the check if anything overlaps; don't assert a fix is correct without re-verifying it. Treat this check as mandatory, not optional polish — visual placement mistakes (a label bleeding into another element, a marker sitting on the icon it's meant to identify) are easy to miss by eye and are the most common failure mode here.

Notes: ask for title/dates first if missing. Walk through each photo and ask for the person's words. If none given (they say "skip," or the photo arrives uncaptioned) → write the note yourself from what's visible in the photo, name the place, add the fact.

Edits: plain-language request → apply and regenerate.
